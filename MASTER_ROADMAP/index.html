    0. Как пользоваться этим документом (прочитать ПЕРВЫМ любой сессии, включая будущего Claude)

    Этот документ — единственный источник правды по архитектурному направлению LEVIATHAN. Он появился после того, как одна сессия в один день трижды развернула видение архитектуры (см. «IMPLEMENTATION LOG» ниже), а отчёт об архитектурном аудите содержал проверяемые ошибки в цифрах (swap 88% вместо реальных 50%, 4 ядра вместо 2, путаница KB/LOC).

    Правила: 1. Не доверяй предыдущим отчётам на слово — прежде чем продолжать любую фазу, перепроверь базовые цифры из §1 командами из §1.1. 2. Ни одна фаза не считается завершённой без независимой проверки — коммит-месседж типа «clean, no tokens» ничего не доказывает (сегодня такой коммит реально содержал живые токены). Каждая фаза ниже имеет раздел «Проверка» с конкретной командой/тестом. 3. Новая архитектурная идея не заменяет этот документ автоматически — если возникает несогласие с планом, править этот файл явно (с обоснованием в Секции 6), а не создавать параллельное «правильное видение» в чате. 4. Секреты — только через /opt/secrets_vault (:8400). Никогда не в markdown/docs/git — сегодня это уже случилось дважды.

    1. Проверенная база на 2026-07-03

    1.1 Как перепроверить

    free -h && nproc && df -h / | tail -1 && uptime
systemctl list-units --type=service --state=running | grep -iE 'leviathan|agent|voice|sav|sport|djvu'
find /opt/leviathan_engine -name '*.py' | xargs wc -l | tail -1

    1.2 Факты на момент написания

    • CPU: 2 ядра (не 4)
    • RAM: 3.8GB всего, ~1.8GB used, ~1.3GB free, ~1.1GB buff/cache
    • Swap: 5.0GB всего, 2.5GB used (50%, не 88%)
    • Disk: 30GB всего, 87% used, 3.7GB свободно
    • Load average: ~2.9–3.6 на 2 ядрах (≈150%/ядро в среднем — реально загружен, хотя и не «88% swap»-уровня)
    • leviathan_engine: 2,692 Python-файла, 412,775 LOC (один из проектов, не все)
    • Живые сервисы (подтверждено systemctl): den4ik-claude, djvu2fb2_web, leviathan-daemon, leviathan-hq, leviathan_agent, leviathan_mcp, savdabot, sport-observer, voicestudio, secrets-vault — 10 сервисов
    • MCP-сервер уже работает (leviathan_mcp.service) — это факт, не план
    • Secrets Vault уже работает (secrets-vault.service, :8400)
    • За этот же день найдены и закрыты: GROQ 401 (provider-mismatch в _groq_pool()/_gemini_pool()), утечка GitHub PAT + приватные ключи в /root/.git, утечка 3× Telegram-токенов + FAL_KEY на публичном GitHub Pages, torch/CUDA-мёртвый груз (4.5GB)

    2. Целевая архитектура (куда идём)

    Слой-модель из третьей версии архитектурного аудита верна по сути и принимается как цель:

    LEVIATHAN ENGINE (платформа)
  Vault · EventBus · Memory · LLM Gateway · Security · Observability · Durable Delivery
          │ API / SDK
          ▼
     3 АГЕНТА (клиенты платформы)
     TRITON · BUFFY · CLAUDE CODE
          │
          ▼
     CAPABILITIES (модули под Triton)
     Arbitrage · Sport · TG Bot · SEO · Pelleto · ...

    Это — горизонт, не завтрашний дедлайн. Ниже — как до него добираться не в хаосе.

    3. ФАЗЫ

    Фаза 0 — Стабилизация (делать первой, низкий риск, высокий эффект)

    Цель: остановить повторение сегодняшних инцидентов до любого архитектурного расширения.

    Задача Почему сейчас Проверка
    pre-push secret-скан (gitleaks) на все git-репо утечка была дважды за один день git push с тестовым ключом в диффе должен блокироваться
    Профилировать реальный источник swap/load (ps aux --sort=-%mem, py-spy) “freebuffd жрёт RAM” — не подтверждено, такого сервиса нет в живом списке Совпадает с §1.1
    Unit-тесты на llm_factory.py/core/key_pool.py Сегодня тут был реальный прод-баг (provider-mismatch) pytest tests/test_llm_factory.py — fail на искусственном mismatch
    CLAUDE.md-правило «не sentence-transformers без --index-url cpu» Уже один раз откатился бы торч-чистку, если бы кто-то переустановил grep правила в CLAUDE.md перед pip install

    Объём: ~4 сфокусированные LLM-сессии, каждая с независимой проверкой после.

    Фаза 1 — Консолидация процессов

    Цель: снизить число всегда-включенных Python-процессов, не строя новый фреймворк с нуля.

    • Использовать уже существующий /opt/leviathan_engine/llm_factory.py как единый LLM-роутер для всех проектов (не писать новый «LLM Gateway» с нуля — он уже есть и сегодня был проверен)
    • Процессы-кандидаты на слияние определяются по результатам профилирования из Фазы 0, не вслепую

    Проверка: free -h до/после — ожидаемый эффект замеряется после Фазы 0, не до.

    Фаза 2 — ADR-002 Kernel

    Спецификация уже готова: docs/context/ADR-002/ADR-002-FINAL.md на leviathan-docs. Не переписывать, только исполнять Фазу 1 из неё (ScenarioPolicy Loader, AgentKernel, DecisionEngine, ExecutionLoop).

    Условие входа: Фаза 0 закрыта (иначе новый kernel будет обрастать на непротестированном LLM-роутинге).

    Проверка: lev-cli run scenario freelance_seo --dry-run должен отдать валидный reasoning_trace.json, не только “Successfully installed” в логах.

    Фаза 3 — Capability Registry

    Оборачиваем уже консолидированные (Фаза 1) модули в BaseCapability. Не раньше — иначе оборачиваем то, что ещё будет переписано.

    Структура (минимальная, без SDK/OpenAPI — это Фаза 4):

    class BaseCapability(ABC):
    capability_id: str
    async def execute(self, params: dict, context) -> Result: ...
    def get_schema(self) -> dict: ...

    Фаза 4 (СПРОЕКТИРОВАТЬ СЕЙЧАС, СТРОИТЬ ПО ТРИГГЕРУ) — Engine как формальная платформа

    Это именно то, о чём ты говоришь: избыточно сейчас, но спроектировано заранее, чтобы не изобретать заново.

    Триггер начать строить: когда появится второй реальный потребитель capability (не Triton), или когда ручная замена компонента через прямой HTTP станет больной точкой (сейчас — нет, прямой вызов через llm_factory достаточен).

    Что проектировать сейчас (документ, не код): - OpenAPI-схема для Vault/Memory/Events (чтобы потом не придумывать на ходу) - Структура leviathan_sdk как Python-пакета (имена классов, не реализация)

    Что НЕ строить до триггера: сам SDK-пакет, миграция Triton на него.

    Фаза 5 (СПРОЕКТИРОВАТЬ СЕЙЧАС, СТРОИТЬ ПО ТРИГГЕРУ) — Multi-server

    Исследовано в этой сессии (веб-поиск, 2026): - A2A (Google) — реальный, зрелый протокол, v1.0 вышел начало 2026, 150+ организаций (Google/Microsoft/AWS/IBM), под Linux Foundation, JSON-RPC 2.0 + SSE + HTTP, подписанные Agent Cards. Если когда-нибудь понадобится агент-к-агент семантика — брать это, не писать своё. - Но твой конкретный сценарий (дешёвый хостинг держит только БД/фронт) — это не agent-задача, это deployment. Готовый ответ 2026 года — Kamal (Basecamp/DHH): SSH-based, на целевом сервере НЕ живёт постоянный демон, оркестрация идёт с твоей машины. Либо Dokploy/Coolify если нужна панель. - Самодельный MAP (WebSocket + свой auth + whitelist команд) — не строить. Это custom RCE-демон на каждом узле, который придётся поддерживать самому — при двух утечках секретов за один день это увеличивает поверхность атаки, не снижает.

    Триггер: когда реально появится второй сервер под конкретный проект. До того — только заметка в этом разделе, код не писать.

    4. СОЗНАТЕЛЬНО НЕ В ПЛАНЕ (и почему)

    Идея Почему не планируется даже на потом
    Capability Marketplace Решает проблему «внешние разработчики делятся модулями» — такой проблемы нет и не предвидится, пока LEVIATHAN обслуживает твои же проекты
    Multi-tenant Engine Решает проблему «чужие клиенты с изоляцией» — есть один оператор (ты)
    Engine Dashboard (Web UI) CLI + Telegram-бот уже закрывают управление; отдельный фронтенд — ещё один процесс на и так загруженном сервере
    Interactive Installer / Scaffolding CLI Имеет смысл, если продукт устанавливают другие люди на свои сервера. Пока устанавливаешь только ты — не нужен

    Если когда-нибудь цель проекта изменится (например, продавать доступ внешним клиентам) — это повод переписать этот раздел явно, а не тихо начать делать.

    5. АНТИ-ХАОС ПРОТОКОЛ (обязателен для любой сессии)

    1. Прежде любого архитектурного решения — прочитать этот файл целиком.
    2. Прежде любого push — скан на секреты (шаблоны команд в §1).
    3. Прежде чем писать “gotovo/clean/fixed” в отчёте — запустить реальный путь кода, не только проверить импорты. Сегодня коммит с текстом «clean, no tokens» содержал живые токены.
    4. Не менять всю архитектурную модель из-за одного сообщения пользователя. Если кажется противоречие с этим документом — обосновать или спросить, не переписывать всё с нуля.
    5. Секреты только через Vault (:8400). Никогда в markdown, docs/, или в URL ремоутов.
    6. После каждой фазы — checkpoint в LEVIATHAN ctx с точным описанием что сделано и чем проверено.

    6. ЖУРНАЛ ИЗМЕНЕНИЙ ЭТОГО ДОКУМЕНТА

    • 2026-07-03: Создан Claude. Заменяет 3 противоречащих видения из одной сессии (6-агентная → 3-агентная+capabilities → Engine-as-platform). Исходная причина: цифры в архитектурном аудите не сошлись с проверкой (swap 88%→реально 50%, 4 ядра→реально 2). Пользователь верно отметил: исполнитель — LLM-сессии, не solo-разработчик; план перестроен под это — избыточное сейчас полностью спроектировано (Фазы 4-5), не отброшено, но строится по триггеру, не по календарю.
