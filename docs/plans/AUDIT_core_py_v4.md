# 📊 Построчный аудит: `agent/core.py` (v4.0, незакоммиченная версия)

**Цель файла:** thin compatibility wrapper над `AgentKernel`/`LLMIntegration` по ADR-002.
**Строк кода:** 270. **Статус:** не закоммичен, заменяет 849-строчную v3.1 (`core.py.v31`).
**Общая оценка: 3/10** — конструктор не работает ни при каких входных данных.

## 🔴 Критические проблемы (блокируют production)

### 1. Строка 211–213 — `StateMachine(initial_state=...)`: гарантированный `TypeError`
```python
self._kernel._fsm = __import__("agent.state", fromlist=["StateMachine"]).StateMachine(
    initial_state=__import__("agent.state", fromlist=["AgentState"]).AgentState.INIT
)
```
**Риск:** `agent/state.py:86` — `class StateMachine: def __init__(self, agent_id: str)`. Параметра `initial_state` не существует. Каждый вызов `LeviathanAgent(...)` упадёт с `TypeError: __init__() got an unexpected keyword argument 'initial_state'` **до** того, как дойдёт до бага #2. Подтверждено чтением реальной сигнатуры, не только предсказанием.
**Это не только в core.py:** идентичная ошибка в `agent/kernel.py:358` — то есть даже правильный вызов `AgentKernel()` (без `__new__`-обхода) сейчас тоже сломан. Корректный пример вызова есть в `agent/improvements/integration.py:37`: `StateMachine(agent_id)`.
**Исправление (не применено):**
```python
self._kernel._fsm = StateMachine(self.agent_id)
```
(и то же в `kernel.py:358`, где есть доступ к `self.agent_id`)

### 2. Строка 226 — `self.state_machine = self._kernel.state_machine`: `AttributeError`
Даже если баг #1 починить, эта строка всё равно упадёт. `AgentKernel` нигде не заводит атрибут `.state_machine` — только `._fsm` (подтверждено `grep`, 0 совпадений на `self.state_machine =` внутри `AgentKernel`). Строка 227 (`self._fsm = self._kernel._fsm`) корректна — а 226, видимо, просто дублирует её с опецаткой в имени.
**Исправление (не применено):** удалить строку 226 либо, если внешний код действительно ждёт `agent.state_machine` — завести алиас `self.state_machine = self._fsm`.

### 3. Строка 210 — `AgentKernel.__new__(AgentKernel)` вместо `AgentKernel(...)`
Обход `__init__` и ручное дозаполнение `_fsm`/`context`/`policy` (строки 210–215) — хрупкий паттерн. Каждый новый атрибут, который появится в `AgentKernel.__init__` (kernel.py) в будущем, молча не попадёт сюда — рассинхронизация будет молчать (не упадёт, просто атрибут будет отсутствовать там, где его ждут).

### 4. Метод `run()` (строки 229–243) не использует `self._kernel` вообще
Докстринг класса: «Delegates to AgentKernel». Фактически `run()` вызывает только `self._llm.run_gemini_loop(...)` напрямую, `self._kernel` ни разу не читается после конструктора. Decision Engine/Role Selector/Replanning/Safety Guards из ADR-002 создаются, но никогда не участвует в выполнении задачи. «Делегирование ядру» заявлено, но не реализовано.

## 🟡 Средние проблемы (техдолг)

- **Строка 79–164 (SYSTEM_PROMPT):** жёстко зашита строка ‚LEVIATHAN AGENT v3.1’ — не обновилась при переходе на v4.0. Абсолютные пути/порты 7 сервисов впечатаны в строку — при переезде сервиса рассинхронизируется бесшумно.
- **Строка 221:** `model_name=kwargs.get("model_name", "gemini-2.0-flash-lite")` — в текущей конфигурации проекта подтверждённая актуальная модель — `gemini-3.1-flash-lite`; здесь старый дефолт, если вызывающий код не передаёт `model_name` явно.
- **Строка 171–182 (`_proto_to_python`):** ветка `return str(obj)` тихо проглатывает нераспознанный тип в строку вместо того, чтобы явно сообщить об ошибке сериализации.
- **Нет try/except вокруг блока строк 210–215:** когда кто-то исправит баги #1/#2, следующая ошибка в этом блоке упадёт сырым stack trace без контекста.

- `Task`/`TaskStep`/`TaskStatus` — аккуратные dataclass'ы, `__post_init__` корректно задаёт `task_id`/`created_at` только при отсутствии.

- Автор явно отметил в docstring, что v3.1 сохранена как `core.py.v31` — откат в один шаг, если потребуется.
