## 🔴 ЧТО ДЕЛАТЬ ПРЯМО СЕЙЧАС

### Шаг 1 — убить осиротевшие freebuff (вернёт ≥1.6GB сразу)
```bash
# Проверить что ни один из них не выполняет активную задачу прямо сейчас
# (открыть каждый tmux attach -t <session> и посмотреть, не без осторожности)
tmux capture-pane -t freebuff-0 -p | tail -20
# если пусто идлит или показывает завершённый чат — убивать

pkill -TERM -f '/root/.config/manicode/freebuff'
sleep 5
pkill -KILL -f '/root/.config/manicode/freebuff'  # если не завершились мягко

# Убить все tmux-сессии группы freebuff/freeb
for s in $(tmux ls -F '#{session_name}' | grep -E '^(freebuff|freeb-|3$|5$)'); do
  tmux kill-session -t "$s"
done

free -h  # проверить эффект
```
**Важно:** оставить одну живую сессию `freebuff` (создана Jul02 15:44) если она активно используется через `herald_worker.py` — проверить `/tmp/herald_worker.lock` перед убийством.

### Шаг 2 — очистить page cache и перепроверить swap
```bash
sync && echo 3 > /proc/sys/vm/drop_caches
free -h
# Критерий: swap < 50%, или хотя бы заметно падает от 86%
```
⚠️ drop_caches не освобождает swap напрямую — если после шага 1 swap всё ещё >70%, выполнить:
```bash
swapoff -a && swapon -a   # принудительно выгружает всё из swap обратно в RAM
# РИСК: если свободной RAM не хватит — может вызвать OOM. Сначала шаг 1, потом это.
```

### Шаг 3 — зафиксировать итоговое состояние
```bash
free -h && uptime && df -h / && ps aux --sort=-%mem | head -10
```
Записать результат в `ctx_checkpoint` и обновить шапку этого файла.

### Шаг 4 — предотвратить повторение (системный фикс, не разовый)
Добавить cron-сторожа на осиротевшие freebuff-процессы (больше 6 часов без связанного tmux-ввода = убить):
```bash
cat > /etc/cron.d/freebuff-reaper << 'EOF'
0 */6 * * * root /opt/leviathan_engine/scripts/reap_orphan_freebuff.sh >> /var/log/freebuff-reaper.log 2>&1
EOF
```
Скрипт `reap_orphan_freebuff.sh` — создать отдельно, логика: если процесс freebuff живёт >6ч И не держит `/tmp/herald_worker.lock` — значит он осиротевший, убить.
