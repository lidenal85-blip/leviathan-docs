## 5. Findings & Issues

### Critical

1. **Hardcoded GitHub token in config** — `/root/.cursor/mcp.json` contains `<REDACTED>` in plain text.
   - **Risk:** Token exposure if config is committed or shared
   - **Fix:** Use environment variable or GitHub CLI auth

2. **MCP server runs as root** — `leviathan_mcp.service` runs as `root` with full system access.
   - **Risk:** Any tool exploit gives complete system compromise
   - **Fix:** Create dedicated `mcp` user with limited permissions

3. **External MCP access with hardcoded Bearer token** — `/opt/pelleto-ai/.mcp.json` uses static token `den4ik1985!`.
   - **Risk:** Token is simple, shared in config file, and used over HTTP (not HTTPS)
   - **Fix:** Use HTTPS, rotate token, use env variable

### Moderate

4. **Two competing MCP server implementations** — `leviathan_mcp_server.py` (FastMCP, modern) and `leviathan_mcp.py` (vanilla MCP, legacy) both serve similar tools.
   - **Issue:** Code duplication, potential inconsistency
   - **Fix:** Deprecate `leviathan_mcp.py` and migrate all clients to the FastMCP version

5. **No rate limiting** — Any MCP client can call `lev_bash` or systemctl without restrictions.
   - **Fix:** Add rate limiting middleware

6. **SQLite-only database support** — No PostgreSQL, MySQL, or MongoDB MCP tools exist.
   - **Fix:** Add database query tools for active databases (PostgreSQL port 5434)

### Minor

7. **Pelleto external MCP access via public IP** — `78.17.24.96:8300` is directly exposed.
   - **Fix:** Use WireGuard/VPN tunnel instead of public exposure

8. **No MCP discovery documentation** — New clients need to guess the MCP endpoint and auth.
   - **Fix:** Add MCP discovery docs to the lev-cli toolset

9. **Context Keeper SQLite WAL** — `/opt/leviathan_engine/logs/lev_context.db` grows unbounded.
   - **Fix:** Add WAL checkpoint cron job
