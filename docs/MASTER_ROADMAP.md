## 0. Как пользоваться этим документом (прочитать ПЕРВЫМ любой сессии, включая будущего Claude)

Этот документ — единственный источник правды по архитектурному направлению LEVIATHAN. Он появился после того, как одна сессия в один день трижды развернула видение архитектуры (см. «IMPLEMENTATION LOG» ниже), а отчёт об архитектурном аудите содержал проверяемые ошибки в цифрах (swap 88% вместо реальных 50%, 4 ядра вместо 2, путаница KB/LOC).

**Правила:**
1. **Не доверяй предыдущим отчётам на слово** — прежде чем продолжать любую фазу, перепроверь базовые цифры из §1 командами из §1.1.
2. **Ни одна фаза не считается завершённой без независимой проверки** — коммит-месседж типа «clean, no tokens» ничего не доказывает (сегодня такой коммит реально содержал живые токены). Каждая фаза ниже имеет раздел «Проверка» с конкретной командой/тестом.
3. **Новая архитектурная идея не заменяет этот документ автоматически** — если возникает несогласие с планом, править этот файл явно (с обоснованием в Секции 6), а не создавать параллельное «правильное видение» в чате.
4. Секреты — только через `/opt/secrets_vault` (:8400). Никогда не в markdown/docs/git — сегодня это уже случилось дважды.

### Фаза 4 (СПРОЕКТИРОВАТЬ СЕЙЧАС, СТРОИТЬ ПО ТРИГГЕРУ) — Engine как формальная платформа

Это именно то, о чём ты говоришь: избыточно сейчас, но спроектировано заранее, чтобы не изобретать заново.

**Триггер начать строить:** когда появится второй реальный потребитель capability (не Triton), или когда ручная замена компонента через прямой HTTP станет больной точкой (сейчас — нет, прямой вызов через `llm_factory` достаточен).

**Что проектировать сейчас (документ, не код):**
- OpenAPI-схема для Vault/Memory/Events (чтобы потом не придумывать на ходу)
- Структура `leviathan_sdk` как Python-пакета (имена классов, не реализация)

**Что НЕ строить до триггера:** сам SDK-пакет, миграция Triton на него.

---

### Фаза 5 (СПРОЕКТИРОВАТЬ СЕЙЧАС, СТРОИТЬ ПО ТРИГГЕРУ) — Multi-server

**Исследовано в этой сессии (веб-поиск, 2026):**
- **A2A (Google)** — реальный, зрелый протокол, v1.0 вышел начало 2026, 150+ организаций (Google/Microsoft/AWS/IBM), под Linux Foundation, JSON-RPC 2.0 + SSE + HTTP, подписанные Agent Cards. **Если когда-нибудь понадобится агент-к-агент семантика — брать это, не писать своё.**
- **Но твой конкретный сценарий (дешёвый хостинг держит только БД/фронт) — это не agent-задача, это deployment.** Готовый ответ 2026 года — **Kamal** (Basecamp/DHH): SSH-based, на целевом сервере НЕ живёт постоянный демон, оркестрация идёт с твоей машины. Либо Dokploy/Coolify если нужна панель.
- **Самодельный MAP (WebSocket + свой auth + whitelist команд)** — не строить. Это custom RCE-демон на каждом узле, который придётся поддерживать самому — при двух утечках секретов за один день это увеличивает поверхность атаки, не снижает.

**Триггер:** когда реально появится второй сервер под конкретный проект. До того — только заметка в этом разделе, код не писать.

## 4. СОЗНАТЕЛЬНО НЕ В ПЛАНЕ (и почему)

| Идея | Почему не планируется даже на потом |
|---|---|
| Capability Marketplace | Решает проблему «внешние разработчики делятся модулями» — такой проблемы нет и не предвидится, пока LEVIATHAN обслуживает твои же проекты |
| Multi-tenant Engine | Решает проблему «чужие клиенты с изоляцией» — есть один оператор (ты) |
| Engine Dashboard (Web UI) | CLI + Telegram-бот уже закрывают управление; отдельный фронтенд — ещё один процесс на и так загруженном сервере |
| Interactive Installer / Scaffolding CLI | Имеет смысл, если продукт устанавливают другие люди на свои сервера. Пока устанавливаешь только ты — не нужен |

Если когда-нибудь цель проекта изменится (например, продавать доступ внешним клиентам) — это повод переписать этот раздел явно, а не тихо начать делать.
